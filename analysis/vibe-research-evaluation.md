## 第一部分：Vibe-Research 仓库评估

### 1.1 项目定位

Vibe-Research 是一个 **个人 AI 投研看板**（Web 应用），主推 A 股、兼看美股/港股。核心理念是"把数据配齐，AI 驱动分析，人不做买卖决策"。技术栈：FastAPI + React 19 + TypeScript + Tailwind。

**它不是推送系统，而是主动查看的看板。** 这一本质差异决定了我们 "汲取精华、不 fork 整体" 的策略。

### 1.2 代码规模

| 模块 | 文件数 | 核心能力 |
|------|--------|----------|
| 后端 `backend/` | 10 个 .py | FastAPI 30+ 端点、AI chat + function calling、MCP server |
| 前端 `frontend/` | 20+ 个 .tsx | 每日复盘、资讯雷达、个股数据、板块中心、持仓、AI 设置 |
| 数据源 a-stock-data | v3.3（内嵌） | 十层架构、40 端点 |
| 全球数据 global-stock-data | v1.0.1（内嵌） | 美港股行情+财务指标 |
| 资讯源 news_sources.json | 108 条分类 RSS | 12 赛道 |
| 板块中心 sectors.json | 19板块 | **2个有效、17个空壳** ⚠️ |

### 1.3 方法论：三层次评估框架

本评估采用以下框架对每个发现进行评分：

| 维度 | 指标 | 权重 |
|------|------|:----:|
| **FEA (Feature Existence Assessment)** | 功能存在性 + 对比用户现有系统优劣 | 必须 |
| **Value-Add Matrix** | 补充进三推送的价值（高/中/低） | 核心 |
| **Architecture Fit Score** | 与用户现有架构的匹配度 | 辅助 |

#### 2.2 高价值、低实现成本 → 强烈建议补充

| # | 发现 | FEA | Value-Add | 成本 | 建议 |
|:-:|------|:---:|:---------:|:----:|:----|
| 1 | **短线情绪模块**（连板梯队/封板率/炸板率/晋级率） | 用户完全缺失。Vibe 用东财 push2ex 四池数据聚合，是量化打板情绪的核心指标 | **高** | **低** | 移植 `market.py:_emotion()` 约 100 行到收盘推送，增加「短线情绪」板块 |
| 2 | **市场情绪机械分档**（冰点→偏弱→中性→偏强→普涨） | 用户有涨跌家数原始数据，但无机械分档 | **中→高** | **极低** | 在现有涨跌家数上加5行分档逻辑，大幅提升可读性 |
| 3 | **赛道 RSS 资讯源**（108 源 × 12 赛道） | 用户有 V2EX/HN/GitHub 等泛科技资讯，但无投资导向的赛道资讯 | **高** | **低** | 复用 `news_sources.json` 中与用户持仓相关赛道（AI/芯片/新能源/医药）约 30 个源 |

#### 2.3 中等价值、需权衡

| # | 发现 | Value-Add | 成本 | 建议 |
|:-:|------|:---------:|:----:|:-----|
| 4 | **成交额 TOP20** | 中（Markdown 长表格影响阅读体验） | 低 | 可加在收盘复盘中，但需精简格式 |
| 5 | **个股研报+一致预期** | 中（基金为主场景需求有限） | 中 | 未来扩展个股研究时核心能力 |
| 6 | **估值历史分位** | 中（ETF/指数投资有意义） | 中 | 未来仪表盘做指数估值时参考 |

### 3. 方向 B：系统设计评估

#### 3.1 前端 UI/UX 设计

Vibe-Research 的前端是一个产品级 Web 应用。以下设计元素值得长期仪表盘参考：

| 设计要素 | 评估 | Architecture Fit Score |
|----------|------|:---------------------:|
| **玻璃暖橙主题** (Glassmorphism + Warm Orange) | 视觉成熟、毛玻璃+暖橙高亮。`GlassCard` 组件统一风格 | **高** — 参考其设计语言 |
| **每日复盘一屏看全** | 大盘→全球→关注股→AI复盘→情绪→短线→TOP20→板块→轮动。层次分明 | **高** — 长期仪表盘布局模板 |
| **AI 即时交互** | 每页面内嵌"问 AI"，自动打包页面数据做 context | **高** — 三推送可扩展为"AI 解读" |
| **赛道资讯+AI提炼** | 12赛道108源，一键提炼今日要点 | **中** — 适合长期规划 |
| **估值历史分位可视化** | ValBand 组件：绿灰红分位带+当前位置标线 | **高** — 可直接参考 |
| **研究笔记系统** | AI 复盘存入 localStorage，Markdown 渲染 | **中** — 不是推送核心需求 |

#### 3.2 架构设计对比

| 架构要素 | Vibe-Research | 用户现有系统 | Fit |
|----------|--------------|-------------|:---:|
| 数据源接入 | 直连 HTTP API，零第三方依赖，分级惰性导入+优雅降级 | 混合（腾讯+天天基金+Yahoo+Weibo） | **中** — 降级模式值得借鉴 |
| 缓存策略 | 全站共享 TTL 缓存（5-30分钟），串行限流+随机抖动防封 | 基本无缓存 | **高** — 长期仪表盘必须加缓存 |
| AI 集成模式 | 三种模式：CLI订阅/API function calling/MCP | 无 AI 分析集成 | **高** — 值得参考的设计模式 |
| 数据与展示分离 | FastAPI + React SPA | 单体脚本 | **中** — 对推送不需要，对仪表盘需要 |
| 隐私设计 | API key 存浏览器，持仓存本地 | 本地文件，无鉴权 | **中** — 个人使用需求不强 |

#### 3.3 Vibe-Research 的硬伤

| 问题 | 严重程度 | 说明 |
|------|:--------:|------|
| **数据源可用性不稳定** | 🔴 高 | 东财接口对大陆住宅 IP 有间歇风控，README 自承认 |
| **板块中心空壳** | 🟡 中 | 19 板块中仅 2 个有实际数据（17 个空壳） |
| **美港股极简** | 🟡 中 | 仅有行情+关键财务，无 K线/技术指标/SEC |
| **无推送能力** | 🔴 高（对我们场景） | 纯 Web 看板，无任何推送机制 |
| **无基金净值** | 🔴 高 | 对用户核心场景致命缺失 |
| **无 KOL 舆情** | 🟡 中 | 缺少微博等社交媒体信号 |

### 4.1 缓存层设计

参考 Vibe 的 `_cached()` 装饰器模式，但针对推送场景简化：

```python
_cache = {}
_CACHE_TTL = {
    'tencent_quote': 60,        # 腾讯行情，1分钟
    'fund_nav': 300,            # 基金净值，5分钟
    'yahoo_quote': 600,         # 外盘，10分钟（数据更新慢）
    'northbound': 60,           # 北向，1分钟
    'sentiment': 120,           # 短线情绪，2分钟
    'kol_weibo': 300,           # 微博，5分钟
}
```

推送场景一天只跑 3 次，缓存的作用是：如果同一脚本内多次请求同一 API，避免重复调用（特别是腾讯批量查询已有此能力）。

### 4.2 新增数据源接入

#### 4.2.1 短线情绪（来自 Vibe `market.py`）

**数据源：** 东财 push2ex 四池数据
**API：** `https://push2ex.eastmoney.com/getTopicZTPool` + `getTopicCTPool`
**接入方式：** 纯 requests + JSON 解析（约 100 行）
**推送位置：** 收盘复盘 `_operation_eval.txt` 底部—新增「短线情绪」板块
**输出格式：**
```
📊 **短线情绪**
| 指标 | 当前 |
|:----|:----:|
| 最高连板 | 5板（XX股份） |
| 封板率 | 67% |
| 炸板率 | 33% |
| 晋级率 | 42% |
| 涨停家数 | 42家 |
```
**注意：** 非交易日和盘中数据为空时跳过

#### 4.2.2 市场情绪机械分档（来自 Vibe `market.py`）

**逻辑（约 5 行）：**
```python
ratio = up_count / (up_count + down_count + 1)
if ratio > 0.7: grade = '普涨 🔴'
elif ratio > 0.55: grade = '偏强 🔴'
elif ratio > 0.45: grade = '中性 🟡'
elif ratio > 0.3: grade = '偏弱 🟢'
else: grade = '冰点 🟢'
```
**推送位置：** 原有涨跌家数数据旁附带

#### 4.2.3 赛道 RSS 资讯（来自 Vibe `news_sources.json`）

**数据源：** 108 个 RSS 源 × 12 赛道
**选择源：** 选取与用户持仓相关的 3 个赛道（AI/芯片、新能源、医药）约 30 个源
**推送位置：** 盘前推送新增「隔夜赛道要闻」板块
**格式：**
```
📡 **隔夜赛道要闻**
AI/芯片：
  ● [标题1](url1) — 来源
  ● [标题2](url2) — 来源
新能源：
  ● [标题3](url3) — 来源
```

### 4.3 设计原则变化

| 原则 | 旧 | 新 |
|------|----|----|
| 数据采集 | 单体函数混文件 | 分层模块（collectors/analyzers/formatters） |
| 缓存 | 无 | TTL 缓存装饰器 |
| 情绪分析 | 无 | 机械分档 + 短线情绪 |
| 资讯 | 泛科技（V2EX/HN） | 投资赛道定向 RSS |
| 趋势展示 | 纯文本 | sparkline Unicode 走势图 |
| AI 集成 | 仅 LLM 读文件输出 | 预留 function calling 接口 |

### 4.4 长期仪表盘规划

参考 Vibe 的看板设计，分阶段建设：

**阶段一（2-4周）：扩展推送内容**
- [ ] 短线情绪模块移植到收盘复盘
- [ ] 市场情绪分档逻辑
- [ ] 赛道 RSS 资讯采集
- [ ] 以上三项通过小版本迭代验证

**阶段二（1-2月）：轻量级静态看板**
- 最简单的形式：`fund_tools.py` 输出 HTML → 上传 R2 作为静态页面
- 参考内容：每日复盘一屏、趋势 sparkline、分组涨跌
- 技术栈：Python 生成 HTML（无需 React）

**阶段三（3月+）：SPA 仪表盘（可选）**
- 需要时参考 Vibe 的架构（FastAPI + React）
- 直接使用 Vibe 的 `GlassCard` CSS 设计语言
- 基金净值追踪（用户独有能力）
