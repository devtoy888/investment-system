# 全量持仓优化 · 数据驱动工作流

> 适用场景：用户要求"重新全量评估持仓"、"给我做持仓优化方案"、"减仓建仓建议"
> 核心原则：所有数据必须独立实时拉取，绝不信任另一个模型的输出数字

## 七步工作要求流

### 步骤1：理解现有方案（只读结构，拒绝数据）

用户可能提供另一个模型/系统的输出报告（HTML/MD/链接）。

```
✅ 正确做法:
  - 提取报告结构（几大部分、决策框架、考量维度）
  - 记录报告中的"建议"类型（减仓/建仓/配置比例）
  - ❌ 绝对不提取其中的数字（涨跌幅、净值、收益率）
  - ❌ 绝对不相信其中的"数据"，全部自己重拉

为什么要拒绝数据:
  - 另一模型的基金净值可能和指数混用（如用中证半导体-19%代替基金实际+101%）
  - 另一模型的数据源可能缓存过时
  - 另一模型的KOL分析可能用预设白名单漏掉赛道
```

### 步骤2：加载持仓数据

```python
# 从 portfolio 文件或持仓脚本读取
# 确认基金代码、成本、组别三要素
fund_codes = [
    ('011613','华夏科创50ETF联接C'),
    ('024418','华夏半导体材料设备ETF联接C'),
    ('014871','大摩科技领先混合C'),
    # ... 全部12支
]
```

⚠️ **代码核对**: 不同脚本可能用不同代码映射（如 025857 vs 021345 都是电网设备ETF联接C，但一个旧一个新）。以 `execute_today_plan.py` 的 PORTFOLIO 字典为准。

### 步骤3：并行拉取所有数据源

在同一轮中发起所有独立的数据拉取：

| 数据源 | 工具 | 用途 |
|:-------|:-----|:------|
| 基金实时估值 | `get_fund_value(code)` | 今日估算涨跌/净值 |
| 基金YTD净值 | `ak.fund_open_fund_info_em(symbol=code, indicator='单位净值走势')` | 全年实际表现 |
| 指数日K线 | `ak.stock_zh_index_daily(symbol='sh000688')` | 板块轮动基准 |
| 板块ETF实时 | `get_tencent_quote('sz159813')` | 今日板块行情 |
| 主任微博 | `get_user_weibos('2014433131', count=50)` | KOL信号判断 |
| 外盘/黄金 (可选) | Yahoo Finance | 全球环境 |

**基金YTD净值注意**：
- AKShare返回列名: `净值日期`, `单位净值`, `日增长率`（不是'date'/'value'）
- 用 `pd.to_datetime(df['净值日期']).dt.date` 转为date对象
- 用 `date(2026,1,1)` 作为YTD起始日
- YTD% = (最新净值 / 年初首日净值 - 1) × 100

### 步骤4：三大分析

#### A. 基金分层分析
对每支基金计算 YTD / 近1月 / 近3月 / 今日估算涨跌 → 排出梯队：
- ⭐ 第一梯队：YTD>50%，是核心持仓
- ⭐ 第二梯队：YTD 20-50%，是配套持仓
- 🟡 辅助：YTD 0-20%，收益有限
- 🟢 承压：YTD<0%，全年亏损

**近1月趋势判断**：YTD好但近1月差 = 动能衰减（要减仓）
**近1月趋势判断**：YTD差但近1月好 = 可能反转（可建仓）

#### B. 重叠分析（同基金经理/同赛道）
- 同一基金经理管理的多支基金：持仓重叠~80%
- 同一公司（大摩）同一方向多支基金：本质是"一个仓位"
- ⚠️ 7支基金看似分散，实际可能只有3个独立方向

#### C. 主任微博分析
拉取全量微博（count=50），按赛道分类：
- 无预设白名单 → 用 open keyword extraction
- 统计频率：市场/风险/科技/长鑫IPO/流动性/存储
- 提取关键语句：主任原话直接引用
- 判断短期方向：高波/触底/谨慎偏多
- 注意主任未提的赛道：未提及=非当前热点

### 步骤5：制定优化方案

#### 减仓原则
1. **同质化基金合并** — 重叠>50%的选一个保留
2. **最弱者先减** — 同组YTD最低+近1月最差的优先
3. **清仓最小规模** — 金额最小且重叠最大的清掉
4. **获利了结** — YTD>80%的减半防回撤
5. **今天大跌不卖** — 急跌反弹后再减，避免恐慌操作

#### 建仓原则
1. **已启动反转** — 近1月板块涨幅最大（如医疗+13%）
2. **主任对冲方向** — 主任配了云厂 → 港股科技
3. **底部布局** — YTD深跌且近1月企稳（如消费-19%）
4. **分批建仓** — 分2-4周，单批≤1/3总分配
5. **单日涨>3%不买** — 等回调

#### 配置结构（参考）
- 科技主线: 40-50%（核心不超过4支）
- 医药防御: 8-12%
- 港股对冲: 5-8%
- 黄金避险: 5-8%
- 消费底部: 3-5%
- 现金保留: 15-20%

### 步骤6：生成报告

**报告必须包含**：
1. 当前持仓全景表（成本/市值/占比/盈亏）
2. 基金真实YTD表现表（YTD/近1月/近3月，标注排名）
3. 重叠分析（同基金经理/同公司）
4. 主任微博分析（频次统计+原话引用）
5. 指数板块环境（各指数YTD+近1月对比）
6. 减仓方案（每支基金的建议操作+理论依据）
7. 建仓方案（标的/金额/分批方式/理论支撑）
8. 优化后配置结构（调整前后对比）
9. 执行计划（哪天做什么）
10. 风险提示（3种情景推演）

**格式执行**：
- 所有数据用markdown表格（不用```代码块）
- 基金涨跌用正负号（+/-）标注
- 主任原话用blockquote
- 数据必须带来源说明
- MD + HTML 同时上传R2

### 步骤7：上传R2

```python
# 上传MD

```

## 常见陷阱

1. **不要用行业指数替代基金净值** — 中证半导体-19% ≠ 半导体材料设备基金+101%，跟踪指数完全不同
2. **基金代码映射核对** — 持仓脚本可能用旧代码，要交叉验证
3. **微博只有近期数据** — API count=50可能只返回7月，6月拿不到是正常的，不要编造
4. **今天大跌减仓是错的** — 要等反弹再执行减仓
5. **观察仓不动** — 小额持仓（<20元）是用户留的观察标的，不要建议清仓
6. **其他模型的报告只借框架，不借数字** — 每个涨跌幅必须自己重算
