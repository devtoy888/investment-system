## 7种报告类型 & Prompt设计

每个prompt由三部分组成：**T+1框架 + 金融理论体系 + 角色/步骤指令**

### 1️⃣ 收盘复盘 (closing) — 6步链

```
角色: 15年经验首席分析师
步骤: ①市场全景定性 → ②板块轮动解构(最强3+最弱3+风格切换)
      ③组合影响分赛道逐基诊断 → ④关键信号识别
      ⑤明日3种情景推演(含概率) → ⑥明日基金操作方向参考
max_tokens: 3500
```

### 2️⃣ 早盘 (morning) — 5步链

```
角色: 首席分析师(开盘前09:00)
步骤: ①隔夜信号传导链 → ②昨日复盘+预测验证
      ③今日3条作战主线 → ④风险清单 → ⑤今日基金关注优先级(关注3+警惕3)
max_tokens: 2000
```

### 3️⃣ 午盘 (noon) — 5步链

```
角色: 首席分析师(半日数据)
步骤: ①上午定位vs昨日预测 → ②半日轮动 → ③量价深度
      ④午后策略调整 → ⑤午后基金操作方向(赛道级)
max_tokens: 2000
```

### 4️⃣ 14:30决策 (decision) — 4步链（最详细）

```
角色: T+1波段操作分析师(收盘前30分钟)
前置: T+1框架 + 金融理论 + 8项数据源声明
步骤: ①全市场画像(含5日趋势定位+新闻驱动)
      ②分赛道多空诊断(含多日趋势+KOL+新闻交叉)
      ③逐基金评分+操作建议(技术分/资金分/趋势分 / 总分 / 操作 / 置信度 / 仓位)
      ④风险检查清单(4项逐项触发)
输出: 每支基金完整表格 | 代码 | 简称 | 赛道 | 技术分 | 资金分 | 趋势分 | 总分 | 操作 | 理由 | 置信度 |
仓位要求: 建仓基金可加仓+仓位建议, 减仓观察需仓位幅度
max_tokens: 2500
```

### 5️⃣ 周度复盘 (weekly) — 4步链

```
角色: 周度组合策略分析师
步骤: ①本周全景 → ②组合归因 → ③KOL聚合(含准确率评估)
      ④下周策略(赛道配比+基金调仓+仓位幅度+赛季配比)
max_tokens: 3000
```

### 6️⃣ 周末外盘 (weekend) — 4步链

```
角色: 首席分析师(周末)
步骤: ①全球市场全景(传导链) → ②持仓赛道影响评估
      ③KOL聚合 → ④周一开盘策略(3种情景+概率+监控维度)
max_tokens: 2500
```

## 共同注入框架

每个提示词开头都注入以下两个框架（约1200字符）：

### T+1操作规则 (511字)
- 15:00前买入/卖出 → 今日净值确认 → 次日到账
- 不能日内交易
- 操作类型定义：可加仓/减仓/持有观望的时间含义
- 建仓期保护(003096/013403永不减仓)
- 时间敏感性(09:00/11:30/14:30不同窗口不同策略)

### 金融投资理论体系 (683字)
- **趋势跟踪**: 单日≠趋势, 需3日确认, 加速/减速/反转信号
- **风险控制**: -8%减半仓, -15%止损, -25%深套不动, 赛道≤50%
- **仓位管理**: 分批建仓, 金字塔加仓(递减), 盈利后再加仓, 留10%现金
- **再平衡**: 目标配比(科技45%/新能源15%/资源15%/黄金10%/医药10%/恒生5%), 偏离>10%触发
- **T+1纪律**: 14:30-15:00决胜窗口, 周末前注意外盘风险

## 数据构建

每个数据构建函数从 `/tmp/fund_data/` 读取当日数据 + 从 `fund_system_data/` 读取历史数据和持仓：

| 数据段 | 来源 | 包含内容 |
|:-------|:-----|:---------|
| 行情数据 | _closing_tables.md / _noon_market.txt 等 | 指数/板块/量价/北向 |
| KOL观点 | _kol_summary.txt / _noon_kol.txt | 主任+小浣熊微博 |
| 多日趋势 | daily-snapshots.jsonl | 5日指数变化表 |
| 持仓盈亏 | daily-snapshots.jsonl | 总成本/市值/盈亏率 |
| 市场新闻 | news_sources.json (RSS) | 6-8条头条 |
| 预测回溯 | predictions.jsonl | 昨日预测+准确率 |
| 完整持仓 | operations/ 目录 | 14支基金成本/天数/赛道分布 |

所有数据段合计约6000-8000字符，**不截断传递**（v1曾经截断data[:300]）。

## 进化引擎演化路径

```
simple_generate() → two_pass_generate(max_out=可调)
  Pass 1: 初稿(draft)     ← max_out tokens
  Pass 2: 自审+评分       ← 600 tokens
  Pass 3: 有条件打磨       ← max_out tokens(如果评分<阈值)
```

**关键修复(2026-07-20)**: 进化引擎原来硬编码 max_tokens=1200，导致v2长输出被截断。
修复后 full_evolution_cycle() 接受 max_tokens 参数，透传到所有pass。所有report_type的max_tokens配置在 generate_v2() 的config字典中。

## 降级链

```
进化引擎 full_evolution_cycle() → 失败fallback到 call_ds() 直接生成
→ 再失败 → None (报告不推送)
```

## 质量验证

`llm_validate.py` 从5个维度自评估(1-10): 数据准确性/逻辑一致性/实用性/简洁性/风险意识
总分 = 5维平均(不是模型输出的和)

**陷阱**: 永远不直接读LLM输出的"总分"——LLM输出的是5维之和(如38)而非均分(7.6)。
评估报告HTML必须内嵌markdown到JS模板字面量，禁止前端fetch()（R2无CORS头）。
