# GB/T 28449-2018 信息安全技术 网络安全等级保护测评过程指南

## 一、标准概述

### 1.1 标准基本信息

| 项目 | 内容 |
|------|------|
| 标准编号 | GB/T 28449-2018 |
| 标准名称 | 信息安全技术 网络安全等级保护测评过程指南 |
| 英文名称 | Information security technology — Evaluation process guide for classified protection of cybersecurity |
| 发布日期 | 2018-12-28 |
| 实施日期 | 2019-07-01 |
| 代替标准 | GB/T 28449-2012 |
| 归口单位 | 全国信息安全标准化技术委员会（SAC/TC 260） |
| 标准性质 | 推荐性国家标准 |

### 1.2 标准定位

GB/T 28449-2018 是网络安全等级保护制度（等保2.0）的核心配套标准之一，与 GB/T 22239-2019《信息安全技术 网络安全等级保护基本要求》、GB/T 25070-2019《信息安全技术 网络安全等级保护安全设计技术要求》和 GB/T 28448-2019《信息安全技术 网络安全等级保护测评要求》共同构成等保2.0的标准体系。

本标准规定了网络安全等级保护测评工作的测评过程，包括测评准备活动、方案编制活动、现场测评活动、报告编制活动四个基本阶段，以及测评过程中的风险控制、文档管理等要求。

在等保2.0标准体系中，GB/T 28449 扮演着"过程总管"的角色——它不定义具体的安全技术要求（那是 GB/T 22239 的职责），也不规定测评的具体判定标准（那是 GB/T 28448 的职责），而是专注于回答一个核心问题：**如何科学、规范、高效地组织和实施一次合格的等级保护测评工作**。

### 1.3 适用范围

本标准适用于：

- **测评机构**：指导第三方测评机构开展网络安全等级保护测评工作，确保测评过程的规范性、一致性和可追溯性。测评机构应将本标准作为内部质量管理体系的核心依据。
- **运营使用单位**：指导网络运营者、使用单位在等级保护定级、整改和自查过程中参考测评过程，帮助其理解测评工作的方法论，从而更好地配合测评工作。
- **主管部门**：为主管部门和监管部门开展监督检查提供参考，使得监督检查工作与正式测评工作在方法论上保持一致。
- **其他相关方**：为系统建设、运维、管理等人员提供测评过程参考，帮助其在系统全生命周期中植入安全考量。

### 1.5 术语和定义

| 术语 | 定义 |
|------|------|
| 等级保护测评 | 对已定级的等级保护对象的安全保护能力是否达到相应等级基本安全要求的检测和评估活动 |
| 测评对象 | 实施等级保护测评过程中所涉及到的物理环境、网络、设备、系统、应用、数据、人员、文档等具体资产 |
| 测评指标 | 用于衡量等级保护对象安全保护能力是否达到相应等级基本安全要求的检查项 |
| 测评单元 | 按照安全类划分的测评指标集合 |
| 测评方法 | 测评人员获取测评证据的具体手段 |
| 测评证据 | 测评过程中通过测评方法获取的能够证明安全措施是否达到要求的各种信息 |
| 测评结论 | 基于测评结果分析得出的对等级保护对象安全保护能力的评价 |

### 2.2 测评过程的基本要求

测评过程应遵循以下基本原则：

1. **客观公正原则**：测评活动应基于客观证据，不受任何利益关系干扰。测评人员应保持独立判断，不得因被测单位的配合程度或客户关系而影响测评结论。
2. **规范性原则**：测评过程应严格按照标准规定的流程和方法开展，确保测评结果的可比性和可追溯性。任何偏离标准的做法都应有充分的理由并记录在案。
3. **保密原则**：对测评活动中获取的信息承担保密义务。包括但不限于被测系统的配置信息、业务数据、安全漏洞信息等。
4. **最小影响原则**：测评活动应尽量减少对被测系统正常运行的影响。对于可能影响业务连续性的测试操作，应事先制定回滚方案和应急预案。
5. **标准性原则**：测评判定应以国家标准为依据。测评人员的主观判断应与标准要求保持一致，避免在判定标准上出现因人而异的情况。
6. **可追溯原则**：所有测评过程和结果均应留下可追溯的记录，使得任何第三方在事后都可以通过记录验证测评工作的充分性。

### 2.3 测评参与角色

| 角色 | 职责说明 | 资质要求 |
|------|----------|----------|
| 测评项目负责人 | 全面负责测评项目的组织、管理、协调和质量控制，对项目最终成果负责 | 具备高级测评师资格，3年以上等保测评经验 |
| 测评组长 | 负责现场测评工作的组织实施和技术把关，审核测评结果的技术正确性 | 具备中级及以上测评师资格，2年以上等保测评经验 |
| 测评工程师 | 执行具体的测评操作，包括访谈、检查、测试等，记录测评证据 | 具备初级及以上测评师资格 |
| 被测单位配合人员 | 提供被测系统信息，协调内部资源配合测评工作开展 | 了解被测系统情况的管理或技术人员 |
| 质量监督人员 | 对测评过程进行独立质量监督和审核，确保过程合规 | 具备内部质量审核资质 |

|------|----------|----------|----------|

## 三、测评准备活动

### 3.1 项目启动

#### 3.1.1 活动目标

测评准备活动是整个测评工作的起点，其质量直接影响后续所有阶段的效果。核心目标包括：

- 充分了解被测系统的基本情况和安全需求
- 建立测评项目组并明确职责分工
- 与被测单位建立有效的沟通协作机制
- 收集编制测评方案所需的基础信息
- 为后续方案编制和现场测评奠定基础

#### 3.1.2 工作内容

**（1）组建测评项目组**

测评机构应根据项目规模、系统复杂度和安全保护等级组建测评项目组，明确各成员职责：

- 测评项目负责人：1名，具备高级测评师资格，对项目全面负责
- 测评组长：1名，具备中级及以上测评师资格，负责技术方案的编制和现场技术把关
- 测评工程师：若干名（根据系统规模，通常3-7人），具备初级及以上测评师资格，负责执行具体测评任务
- 质量监督人员：1名（可由项目负责人兼任，但应保持质量审核的独立性）

**人员配置参考**：

| 系统保护等级 | 测评工程师人数 | 预计周期 | 说明 |
|--------------|---------------|----------|------|
| 第一级 | 2-3人 | 2-3个工作日 | 系统简单，测评项少 |
| 第二级 | 3-5人 | 5-8个工作日 | 系统中等复杂度 |
| 第三级 | 4-7人 | 10-15个工作日 | 系统较复杂，测评项多 |
| 第四级 | 6-10人以上 | 15-20个工作日以上 | 系统非常复杂，要求最严格 |

**（2）明确测评目标**

与被测单位沟通，明确本次测评的目标、范围和依据：

- **测评目标**：确定是首次定级测评、定期复评（等保要求三级系统每年一次测评、四级系统每半年一次）、还是整改后的补充测评
- **测评范围**：明确被测系统的边界（网络边界、物理边界、应用边界），确定涉及的网络区域、物理位置、业务应用范围
- **测评依据**：GB/T 22239-2019 基本要求中对应等级的安全要求作为评判基准
- **测评类型**：确认是通用要求测评，还是包含云计算/移动互联/物联网/工控/大数据等扩展要求

**（3）签订测评协议**

双方签订测评服务协议或合同，明确：

- 测评工作内容和范围（含是否包含渗透测试等附加服务）
- 双方责任和义务（包括配合义务、保密义务等）
- 保密要求和数据安全要求
- 时间安排和里程碑节点
- 交付成果清单（测评报告、问题清单、整改建议等）
- 费用和支付方式
- 争议解决机制

### 3.2 信息收集与分析

#### 3.2.1 信息收集内容

测评机构应通过问卷调查、资料查阅、人员访谈等方式收集以下信息：

**（1）系统基本信息**

| 信息类别 | 具体内容 | 收集方式 |
|----------|----------|----------|
| 系统定级信息 | 系统名称、安全保护等级、定级报告、备案证明 | 查阅定级文档 |
| 系统概述 | 业务描述、网络拓扑、系统架构、服务范围、用户规模 | 问卷+访谈 |
| 网络结构 | 网络拓扑图、网络区域划分、VLAN划分、边界设备清单 | 查阅设计文档 |
| 设备清单 | 服务器、网络设备、安全设备、存储设备等型号和数量 | 查阅资产清单 |
| 软件清单 | 操作系统、数据库、中间件、应用系统及版本 | 查阅配置文档 |
| 安全措施 | 已部署的安全产品清单、安全策略概要 | 访谈+查阅 |
| 管理制度 | 安全管理机构设置、安全管理制度体系、操作规程 | 查阅制度文件 |
| IP地址规划 | IP地址段分配、内外网划分、管理网段 | 查阅地址规划 |
| 业务应用 | 关键业务应用描述、数据流图、对外服务接口 | 访谈+文档 |
| 运维模式 | 自运维/外包运维、运维工具、变更管理流程 | 访谈 |

**（2）物理环境信息**

- 机房的物理位置（建筑物名称、楼层、房间号）和周边环境（是否存在水患、火灾、电磁干扰等风险源）
- 机房面积、设备布局、机柜数量和分布
- 门禁系统类型（刷卡/指纹/人脸识别）和管理方式
- 视频监控系统覆盖范围和录像保存时间
- 消防系统类型（气体灭火/水喷淋等）和联动机制
- 温湿度控制设备的类型和冗余情况
- 电力供应方式：市电接入路数、UPS容量和后备时间、发电机配置
- 防雷接地系统的情况

**（3）管理文档信息**

- 安全管理机构和岗位设置文件
- 安全管理制度体系文件（总体方针、各类管理制度）
- 人员安全管理制度（录用、离职、考核、培训等）
- 系统建设管理文档（方案设计、采购、开发、测试、验收等）
- 系统运维管理文档（日常运维、变更管理、备份恢复等）
- 应急响应预案和演练记录
- 安全事件处理记录
- 安全检查与审计记录

#### 3.2.2 信息分析方法

对收集到的信息进行系统分析，为测评方案编制提供依据：

**（1）定级信息核实**

- 检查定级报告是否在有效期内
- 核实业务信息安全等级和系统服务安全等级的确定是否合理
- 确认最终安全保护等级的确定依据
- 验证定级结果是否与系统实际的业务重要性和安全需求匹配

**（2）系统边界识别**

- 明确被测系统的网络边界（与外部网络的接口）
- 明确资产边界（哪些设备、系统、数据属于被测系统）
- 明确安全边界（安全措施的有效防护范围）
- 识别被测系统与其他系统的互联关系和数据交互

**（3）重点保护对象确定**

- 识别核心业务数据和关键业务功能
- 确定关键网络设备和安全设备
- 识别重要的服务器和应用系统
- 确定重要的安全区域（如核心数据库区）

**（4）安全风险预判**

- 根据网络拓扑初步分析可能存在的安全薄弱点
- 根据已部署安全产品分析防护能力缺口
- 根据管理制度情况预判管理层面的可能问题
- 为测评方案中重点关注领域提供参考

#### 3.2.3 信息收集的输出物

| 输出物 | 说明 | 用途 |
|--------|------|------|
| 被测系统信息调查表 | 详细记录系统基本信息和分析结果 | 方案编制基础 |
| 系统定级报告（复印件） | 确认定级情况的法定文件 | 确定测评等级 |
| 网络拓扑图 | 了解网络结构和边界 | 确定测评对象 |
| 安全管理制度清单 | 明确管理文档审查范围 | 文档审查依据 |
| 资产清单 | 系统软硬件资产详细列表 | 抽样依据 |
| 信息分析报告 | 对收集信息的综合分析和风险预判 | 方案编制参考 |

### 3.3 工具和表单准备

#### 3.3.1 测评工具准备

根据被测系统情况准备相应的测评工具，并进行功能验证：

| 工具类别 | 用途 | 典型工具示例 | 验证方法 |
|----------|------|-------------|----------|
| 漏洞扫描工具 | 发现系统和应用漏洞 | Nessus Professional、OpenVAS、绿盟RSAS | 搭建测试环境验证扫描功能 |

- 所有测评工具在使用前应经过功能验证，确保正常工作

- 工具检测过程中产生的临时数据应在测评结束后清理
- 测评笔记本电脑应做好安全加固（全盘加密、杀毒软件等）

- 测评项目组全体成员

- 被测单位配合人员

## 四、方案编制活动

### 4.1 测评对象确定

#### 4.1.1 测评对象确定原则

测评对象是指测评过程中需要检查、测试和评估的具体资产，包括物理环境、网络设备、安全设备、服务器、终端、应用系统、数据和管理文档等。确定测评对象时应遵循以下原则：

1. **重要性原则**：优先选择对系统安全起关键作用的资产。核心业务服务器、核心网络设备、关键安全设备等必须纳入测评对象。
2. **代表性原则**：同一类别的大量资产中，选取具有代表性的样本进行抽样测评。抽样时应考虑不同型号、不同配置、不同位置的资产均有代表。
3. **全面性原则**：覆盖安全要求的所有层面（技术+管理），不能出现"测评盲区"。
4. **适当性原则**：在保证测评有效性的前提下合理控制测评工作量，避免过度测评或测评不足。
5. **可操作性原则**：选取的测评对象应具备实施测评的条件，如设备在线可访问、有配合人员等。

#### 4.1.2 测评对象类别及抽样策略

| 对象类别 | 具体内容 | 抽样策略 | 最少抽样数（三级系统） |
|----------|----------|----------|------------------------|
| 物理环境 | 机房、配电间、配线间、介质库 | 全部检查，不抽样 | 全部 |
| 核心网络设备 | 核心交换机、核心路由器 | 全部检查 | 全部 |
| 一般网络设备 | 接入交换机、汇聚交换机 | 同类抽样，覆盖不同型号 | 不少于2台或20%（取大） |
| 网络安全设备 | 防火墙、IDS/IPS、WAF、堡垒机、网闸、VPN | 全部检查，安全设备不抽样 | 全部 |
| 服务器-核心 | 核心数据库服务器、核心应用服务器 | 全部检查 | 全部 |
| 服务器-一般 | 一般应用服务器、文件服务器、测试服务器 | 按功能类型抽样 | 每种功能至少1台 |
| 终端设备 | 管理终端、业务终端、运维终端 | 按用途抽样 | 每种用途至少1台 |
| 应用系统 | 核心业务应用、办公系统、外网服务 | 全部检查 | 全部 |
| 数据库系统 | Oracle、MySQL、SQL Server等 | 全部检查 | 全部 |
| 中间件 | Tomcat、WebLogic、Nginx、IIS等 | 全部检查 | 全部 |
| 数据资源 | 业务数据、配置数据、审计日志、备份数据 | 按数据类型抽样 | 每类至少1份样本 |
| 管理文档 | 制度文件、操作规程、记录表单 | 全部检查，不抽样 | 全部 |
| 人员 | 安全管理员、系统管理员、网络管理员、审计员、操作员 | 按岗位抽样 | 每岗位至少1人 |

#### 4.1.3 测评对象选择清单

编制《测评对象选择清单》，为每个测评对象建立详细档案：

| 字段 | 说明 | 示例 |
|------|------|------|
| 测评对象编号 | 唯一编号，用于全过程追踪 | OBJ-001 |
| 测评对象名称 | 设备名称或系统名称 | 核心防火墙-01 |
| 测评对象类别 | 按上述分类 | 网络安全设备 |
| 品牌型号 | 设备品牌和型号 | 华为 USG6650 |
| 软件版本 | 操作系统/固件版本 | V500R001 |
| 物理位置 | 所在机房/机柜 | A座3楼核心机房C03柜 |
| IP地址/管理地址 | 管理访问地址 | 10.1.1.1 |
| 业务重要性 | 关键/重要/一般 | 关键 |
| 选择理由 | 为什么选入测评范围 | 核心边界防护设备 |
| 负责人 | 被测单位该资产负责人 | 张三 |
| 可测评时间窗口 | 允许开展测评的时间 | 工作日18:00-次日6:00及周末 |

### 4.2 测评指标确定

#### 4.2.1 测评指标来源

测评指标是判断安全措施是否达到要求的具体检查项，来源于 GB/T 28448-2019《信息安全技术 网络安全等级保护测评要求》中对应等级的安全要求。

#### 4.2.2 测评指标确定方法

**（1）等级对应**

根据被测系统的安全保护等级，从 GB/T 28448 中选择对应等级的测评指标：

| 安全保护等级 | GB/T 28448 对应章节 | 典型指标数量 | 保护能力要求 |
|--------------|---------------------|-------------|-------------|
| 第一级 | 第一级安全要求 | 约50项 | 自主保护 |
| 第二级 | 第二级安全要求 | 约135项 | 指导保护 |
| 第三级 | 第三级安全要求 | 约211项 | 监督保护 |
| 第四级 | 第四级安全要求 | 约260项 | 强制保护 |

**（2）通用要求与扩展要求的组合**

等保2.0的安全要求分为通用安全要求和扩展安全要求两个层次：

- **通用安全要求（必选）**：所有等级保护对象都必须满足的基本安全要求，涵盖十大安全类。无论被测系统的类型如何，这部分都是必测内容。
- **扩展安全要求（按需选择）**：针对特定类型等级保护对象的额外安全要求。根据被测系统实际情况，选择适用的扩展要求类别：

| 扩展要求类型 | 适用场景 | 扩展指标数量（三级） |
|--------------|----------|---------------------|
| 云计算安全扩展要求 | 系统部署在云计算环境中或本身为云平台 | 约45项 |
| 移动互联安全扩展要求 | 系统提供移动端接入或业务涉及移动终端 | 约30项 |
| 物联网安全扩展要求 | 系统包含物联网感知层或IoT设备 | 约35项 |
| 工业控制系统安全扩展要求 | 系统属于工业控制系统 | 约40项 |
| 大数据安全扩展要求 | 系统涉及大数据平台或大规模数据处理 | 约35项 |

**（3）不适用项的判定与处理**

对于被测系统中不适用的测评指标，应在方案中逐项说明不适用的理由。不适用项判定原则：

- **技术不适用**：系统中确实不存在对应的技术组件（如无无线网络的系统不需要测评无线安全要求）
- **环境不适用**：系统部署环境天然规避了某个风险（如部署在专用机房的系统不需要测评"机房选址"中的部分项）
- **等效替代**：采用了与标准要求等效的其他安全措施（需提供等效性论证）
- **记录要求**：所有不适用项必须在测评方案中明确标注，并在测评报告中予以说明

**不适用项统计要求**：如果某个安全类中不适用项的比例超过30%，应分析原因并在报告中重点说明。

#### 4.2.3 测评指标的类型划分

| 指标类型 | 对应安全类 | 典型测评内容 | 测评方法侧重 |
|----------|-----------|-------------|-------------|
| 安全物理环境 | 物理安全 | 机房选址、门禁、监控、温湿度、电力、防火防水等 | 实地查验+文档审查 |
| 安全通信网络 | 通信网络安全 | 网络架构、通信传输、可信验证 | 配置核查+工具检测 |
| 安全区域边界 | 区域边界安全 | 边界防护、访问控制、入侵防范、安全审计 | 配置核查+工具检测 |
| 安全计算环境 | 计算环境安全 | 身份鉴别、访问控制、安全审计、入侵防范、数据完整性等 | 多种方法综合 |
| 安全管理中心 | 安全管理 | 系统管理、审计管理、安全管理、集中管控 | 配置核查+访谈 |
| 安全管理制度 | 管理制度 | 安全策略、管理制度、操作规程 | 文档审查+访谈 |
| 安全管理机构 | 管理机构 | 岗位设置、人员配备、授权审批 | 文档审查+访谈 |
| 安全管理人员 | 人员安全 | 人员录用、离岗、培训、考核、外部人员管理 | 文档审查+访谈 |
| 安全建设管理 | 建设管理 | 定级备案、方案设计、采购、开发、测试、验收 | 文档审查 |
| 安全运维管理 | 运维管理 | 环境管理、资产管理、介质管理、设备维护、漏洞和风险管理、备份恢复等 | 文档审查+配置核查+访谈 |

### 4.3 测评实施方案编制

#### 4.3.1 测评方案的内容

测评方案是指导现场测评活动的技术文件，也是测评工作质量保证的基础。一份完整的测评方案应包含以下章节：

**第一章：项目概述**
- 测评项目背景和委托情况
- 被测系统基本情况（名称、用途、用户规模、数据量级等）
- 安全保护等级和定级时间
- 测评目标和范围
- 测评依据（列出所有参考标准）

**第二章：测评对象**
- 测评对象选择原则说明
- 测评对象详细清单（含选择理由）
- 测评对象分布图（网络拓扑标注）

**第三章：测评指标**
- 测评指标矩阵（安全类 × 测评对象）
- 测评指标详细列表（含指标编号、名称、来源、测评方法）
- 不适用项清单及理由说明
- 扩展要求指标（如适用）

**第四章：测评实施内容**
- 各测评指标的具体检查方法描述
- 测评操作步骤和预期结果
- 测评工具使用计划
- 证据收集要求

**第五章：测评实施计划**
- 总体时间安排（起止日期）
- 每日详细工作计划（上午/下午各安排什么内容）
- 人员分工和职责
- 被测单位配合需求（人员、文档、场地、网络接入等）
- 风险控制措施和应急预案

**第六章：附录**
- 测评表单模板
- 测评工具清单
- 沟通计划

#### 4.3.2 测评方案的评审

测评方案编制完成后，应经过多级评审流程：

**评审层级和重点**：

| 评审层级 | 评审人 | 评审重点 |
|----------|--------|----------|
| 一级：自审 | 方案编制人 | 内容完整性、数据准确性、格式规范性 |
| 二级：组内评审 | 测评项目组成员 | 技术路线合理性、对象选择充分性、方法可行性 |
| 三级：技术评审 | 机构技术负责人 | 技术方案的合规性、判定标准的一致性、工具选择的合理性 |
| 四级：质量评审 | 质量管理人员 | 过程合规性、文档规范性、风险控制充分性 |
| 五级：客户确认 | 被测单位 | 范围合理性、时间可行性、配合可操作性 |

#### 4.3.3 测评方案确认

测评方案经内部评审通过后，应正式提交被测单位进行确认：

- 确认测评范围和对象是否合理，是否需要增加或减少测评对象
- 确认测评时间安排是否可行，是否与业务高峰期冲突
- 确认配合需求和风险控制措施是否适当
- 被测单位对方案提出修改意见的，测评机构应认真评估并协商调整
- 方案确认后，双方签字确认测评方案作为后续工作的依据
- 如现场测评过程中需要调整方案，应通过变更程序并经双方确认

## 五、现场测评活动

### 5.1 现场测评启动

#### 5.1.1 首次会议

现场测评工作正式开始前，应召开首次会议，这是现场测评的正式起点。

**参会人员**：

- 测评项目组全体成员
- 被测单位信息安全负责人
- 被测单位IT运维负责人
- 被测单位指定的配合人员
- 被测单位相关系统管理员、网络管理员
- （必要时）被测单位法务人员或合规人员

**会议议程（详细）**：

1. **人员介绍**：双方参会人员相互介绍，确认各自角色和职责
2. **目标重申**：测评组长重申本次测评的范围、目标和标准依据
3. **计划通报**：通报详细的现场测评工作计划，包括每日工作内容、需要配合的人员和时间
4. **配合确认**：确认被测单位为测评工作提供的配合条件：
   - 网络接入方式（临时IP、VLAN划分等）
   - 办公场所和设施
   - 文档资料查阅方式
   - 系统访问权限（如需要创建临时账号）
   - 机房/物理环境进入许可
   - 访谈时间和人员安排
5. **风险告知**：说明测评过程中可能产生的风险和已采取的控制措施，获取被测单位的理解和配合
6. **工具授权**：提交测评工具清单和使用范围说明，获取被测单位书面授权
7. **保密承诺**：测评组成员签署保密承诺书，承诺对测评中获取的信息保密
8. **沟通机制**：确认双方沟通联系人、沟通方式和频率
9. **应急联络**：确认紧急情况下的联系方式和应急处理流程
10. **会议纪要**：形成首次会议纪要并由双方签字

#### 5.1.2 测评环境确认

#### 5.2.1 访谈（Interview）

**（1）方法概述**

访谈是通过与被测系统相关人员进行有目的的面对面交流，了解系统安全管理情况、技术措施落实效果和人员安全意识的测评方法。访谈是非技术性测评方法中最为灵活、信息量最大的一种。

**（2）访谈对象和重点内容**

| 访谈对象 | 访谈重点内容 | 典型问题示例 |
|----------|-------------|-------------|
| 信息安全主管 | 安全管理体系建设情况、安全工作规划和实施、安全事件处理 | "请介绍贵单位的安全管理组织架构和职责分工" "最近一次安全事件是如何处置的？" |
| 系统管理员 | 系统安全配置、账户和权限管理、补丁管理策略、备份恢复流程 | "请演示一下日常的账户创建和权限分配流程" "系统补丁是如何获取、测试和部署的？" |
| 网络管理员 | 网络拓扑和区域划分、访问控制策略、网络监控和日志审计 | "请介绍网络区域的划分原则和访问控制策略" "网络异常流量是如何监测和处理的？" |
| 数据库管理员 | 数据库安全配置、访问控制、加密措施、备份策略 | "数据库的访问权限是如何分级管理的？" "请说明数据库备份策略和恢复测试情况" |
| 应用系统管理员 | 应用安全、用户权限管理、应用日志、开发安全 | "应用系统的用户权限是如何管理和定期审计的？" "应用上线前是否经过安全测试？" |
| 安全审计员 | 审计策略配置、审计记录管理、审计结果分析和利用 | "审计日志保存多长时间？是否定期进行审计分析？" "是否发现过异常审计记录？如何处理？" |
| 普通终端用户 | 安全意识、日常操作规范、安全制度知晓情况 | "是否参加过安全培训？培训内容是什么？" "收到可疑邮件会如何处理？" |
| 运维值班人员 | 日常监控内容、值班记录、应急预案熟悉度 | "请描述一下您值班期间的安全监控内容" "如果发现服务器异常，您的处理步骤是什么？" |

**（3）访谈实施要求**

- **准备充分**：访谈前应根据测评指标编制详细的访谈提纲，明确需要获取的信息。提纲应覆盖所有需要通过访谈验证的测评项。
- **环境适宜**：访谈应在安静、独立的环境中进行，避免干扰，鼓励被访谈人坦诚交流。
- **时间合理**：单次访谈时间控制在30-60分钟，避免被访谈人疲劳。重要岗位可安排多次访谈。
- **完整记录**：详细记录访谈时间、地点、对象、内容和结论。关键信息应逐字记录。
- **签字确认**：访谈记录应由被访谈人审阅后签字确认，避免后续争议。
- **交叉验证**：同一安全措施应通过访谈不同岗位的人员进行交叉验证。例如，备份制度既要访谈系统管理员，也要查阅备份记录，还要实际检查备份文件。
- **行为观察**：访谈过程中可以结合行为观察，例如请管理员现场演示某项操作流程。

**（4）访谈记录表标准格式**

| 记录项目 | 内容 |
|----------|------|
| 访谈编号 | FT-YYYYMMDD-NNN（如FT-20260711-001） |
| 访谈人 | 测评工程师姓名 |
| 记录人 | 负责记录的人员姓名 |
| 被访谈人 | 姓名/所属部门/职务/联系方式 |
| 访谈时间 | YYYY-MM-DD HH:MM - HH:MM |
| 访谈地点 | 具体的房间或位置 |
| 关联测评指标 | 本次访谈涉及的测评指标编号列表 |
| 被访谈人角色 | 安全主管/系统管理员/网络管理员/... |
| 访谈提纲 | 预先准备的访谈问题列表 |
| 访谈过程记录 | 按问答形式详细记录访谈内容 |
| 访谈结论 | 对获取信息的总结性评价 |
| 证据索引 | 关联的其他证据编号 |
| 被访谈人签字 | |
| 访谈人签字 | |

#### 5.2.2 文档审查（Document Review）

**（1）方法概述**

文档审查是对被测系统的安全管理制度、操作规程、记录文件等进行系统性检查，以验证安全管理措施是否完善、规范、有效。文档是管理痕迹的物化体现，文档审查是验证管理安全要求的主要方法。

**（2）审查文档类别和审查要点**

| 文档类别 | 主要文档名称 | 审查要点 |
|----------|-------------|----------|
| 顶层方针 | 信息安全总体方针、安全策略文件 | 是否有正式的总体方针文件、是否经最高管理层批准发布、是否定期评审更新 |
| 管理制度 | 机房管理规定、设备管理规定、网络安全管理规定、应用系统安全管理规定、数据安全管理规定、用户管理规定 | 是否覆盖所有安全管理领域、是否经过正式审批发布、内容是否与实际情况相符 |
| 操作规程 | 系统操作手册、备份操作SOP、应急操作SOP、变更操作流程 | 操作步骤是否清晰可执行、是否有安全操作要求、是否有版本控制 |
| 岗位文件 | 岗位职责说明书、授权书、保密协议 | 安全岗位是否明确、职责是否清晰、是否有正式的授权记录 |
| 运维记录 | 日常运维日志、值班记录、巡检记录、备份记录、变更记录 | 是否连续完整、内容是否真实可查、是否有审批签字 |
| 审计记录 | 安全审计日志、审计分析报告、审计问题整改记录 | 审计是否覆盖要求范围、是否有分析利用、问题是否闭环处理 |
| 应急文档 | 应急预案、应急演练计划、演练报告、应急事件处理记录 | 预案是否覆盖主要场景、是否定期演练、演练是否有总结改进 |
| 培训文档 | 安全培训计划、培训教材、签到记录、考核记录 | 培训是否覆盖所有相关人员、是否有考核、培训内容是否更新 |
| 建设文档 | 系统设计方案、安全设计方案、测试报告、验收报告、定级报告 | 建设过程是否考虑安全、是否有安全测试、验收是否包含安全项 |
| 外包管理 | 外包服务协议、外包人员保密协议、外包服务考核记录 | 是否明确安全责任、是否有保密要求、是否有监督考核 |

**（3）文档审查方法**

- **完整性检查**：对照标准要求的安全管理领域，检查文档体系是否覆盖全面
- **适宜性检查**：文档内容是否与单位的实际情况相符，是否具备可操作性
- **有效性检查**：文档规定是否得到了实际执行，通过现场访谈和实物检查进行验证
- **时效性检查**：文档是否有版本标识、发布日期、审批记录，是否定期更新
- **合规性检查**：文档内容是否符合国家和行业相关要求

| 审查日期 | |

#### 5.2.3 配置核查（Configuration Verification）

**（1）方法概述**

配置核查是通过登录设备或系统，查看其配置信息和参数设置，验证安全配置是否符合等级保护要求的方法。配置核查是技术安全测评中最基础、最常用的方法。

**（2）核查方式和适用场景**

| 核查方式 | 操作方法 | 优点 | 缺点 | 适用场景 |
|----------|----------|------|------|----------|
| 人工登录核查 | 通过SSH/RDP/Console等登录设备，查看配置文件 | 灵活全面、适用所有设备 | 效率低、依赖个人能力、结果标准化差 | 少量设备、需要深度判断的场景 |
| 自动化工具核查 | 使用等保检查工具箱、CIS-CAT等自动化工具 | 效率高、结果标准化、可批量操作 | 可能无法适配所有设备、需要工具授权 | 大量同类设备、适合做基线核查 |
| 配置导出分析 | 导出设备配置文件（如show run），离线分析比对 | 不占用设备资源、可深度分析 | 需要导出权限、不能观察运行状态 | 防火墙、路由器、交换机 |
| 管理界面查看 | 通过Web管理界面查看配置 | 直观、不需要命令行技能 | 部分配置在GUI上不显示、截图效率低 | 简单设备、应用系统 |

**（3）常见核查对象和关键核查内容**

| 核查对象 | 关键核查内容 |
|----------|-------------|
| **网络设备**（路由器/交换机） | 远程管理（SSH v2，禁用Telnet）；管理IP限制（ACL限制管理来源IP）；SNMP配置（v3，community string非默认）；日志配置（指向日志服务器，含时间戳）；认证授权（AAA/TACACS+/RADIUS）；端口安全；BPDU Guard/Root Guard |
| **防火墙/安全网关** | 默认拒绝策略；最小化开放端口；会话超时配置；管理接口限制；规则库版本和更新策略；日志记录和远程发送；高可用性配置 |
| **操作系统（Linux）** | 口令策略（/etc/login.defs、PAM模块）；账户管理（禁用root远程登录、删除无用账户）；文件权限（关键文件权限符合基线）；审计配置（auditd服务状态和规则）；SSH配置（禁用root登录、协议版本2）；防火墙（iptables/firewalld规则）；时钟同步（NTP配置） |
| **操作系统（Windows）** | 口令策略（密码长度、复杂度、过期时间）；账户策略（账户锁定、Administrator重命名）；审计策略（登录事件、对象访问等）；用户权限分配；服务配置（禁用不必要的服务）；Windows防火墙规则；补丁更新策略；远程桌面限制 |
| **数据库系统** | 认证方式（禁止弱认证）；审计配置（开启审计、审计范围）；访问控制（最小权限原则）；加密配置（传输加密、存储加密）；默认账户处理（禁用或修改默认账户）；监听地址限制 |
| **中间件/Web服务器** | 版本信息隐藏；默认页面和示例应用删除；管理界面访问限制；SSL/TLS配置（禁用弱协议和弱加密套件）；目录列表禁用；HTTP安全头配置；日志配置 |
| **应用系统** | 认证机制（密码强度、多因素认证）；会话管理（超时设置、并发会话限制）；访问控制粒度；密码重置流程安全性；日志功能（操作日志、登录日志）；文件上传限制；防暴力破解机制 |
| **虚拟化平台** | 管理网络隔离；管理接口访问控制；虚拟机隔离；资源限制；日志和审计配置；模板安全加固 |

#### 5.2.4 工具检测（Tool Testing）

**（1）方法概述**

工具检测是使用专业安全检测工具对被测系统进行自动化安全扫描和测试，以发现系统存在的安全漏洞、配置缺陷和潜在风险。工具检测是技术测评中效率最高、覆盖面最广的方法，但需要与被测单位充分沟通并获得授权。

**（2）检测类型详解**

| 检测类型 | 检测目的 | 典型工具 | 输出成果 | 风险等级 |
|----------|----------|----------|----------|----------|
| 漏洞扫描 | 发现操作系统、应用软件已知安全漏洞 | Nessus、OpenVAS、绿盟RSAS | 漏洞清单（含CVSS评分） | 低-中 |
| 端口扫描 | 发现开放的网络端口和运行的服务 | Nmap、Masscan | 开放端口和服务列表 | 低 |
| Web漏洞扫描 | 检测Web应用安全漏洞（SQL注入、XSS等） | Burp Suite、AppScan、AWVS | Web漏洞清单 | 中 |
| 弱口令检测 | 检测系统和应用是否存在弱口令 | Hydra、Medusa、自研脚本 | 弱口令账户列表 | 中 |
| 配置合规扫描 | 自动化检查设备和系统的安全配置基线 | CIS-CAT、等保工具箱 | 配置合规报告 | 低 |
| 协议安全分析 | 分析网络通信协议是否存在安全风险 | Wireshark、tcpdump | 协议分析报告 | 低-中 |
| 渗透测试 | 模拟真实攻击验证安全防护有效性 | Metasploit、Cobalt Strike | 渗透测试报告 | 高 |
| 恶意代码检测 | 检测系统是否存在病毒、木马等恶意代码 | 杀毒软件、专杀工具 | 恶意代码检测报告 | 低 |
| 代码审计 | 检查应用源代码是否存在安全缺陷 | Fortify、SonarQube | 代码审计报告 | 低 |
| 无线安全检测 | 检查无线网络的安全配置 | Aircrack-ng、Kismet | 无线安全报告 | 中 |

**（3）工具检测操作规范**

进行工具检测应严格遵守以下操作规范：

1. **授权先行**：任何工具检测操作必须获取被测单位书面授权，明确检测范围、时间和限制
2. **范围限定**：严格按照授权范围进行检测，不得超出范围和权限
3. **时间控制**：可能影响系统性能的扫描应在非业务高峰期进行（如深夜或周末）
4. **先测试后实施**：对未知影响的检测操作，应先在测试环境验证后再用于生产系统
5. **参数谨慎**：扫描参数应从温和模式开始（如仅扫描不利用漏洞），根据需要逐步提高强度
6. **实时监控**：在检测过程中实时监控被测系统的运行状态，一旦发现异常立即停止
7. **数据保护**：检测过程中获取的任何数据都应妥善保护，检测完成后清理所有临时数据
8. **操作记录**：记录所有工具检测的操作过程、时间、范围和结果

**（4）渗透测试的特别要求**

渗透测试是工具检测中风险最高的一种，需要特别注意：

- **明确授权**：渗透测试需要单独的、更高级别的授权，明确授权范围和测试深度
- **独立操作**：渗透测试应在相对独立的环境中进行，避免对生产系统造成不可逆影响
- **紧急联系**：建立24小时紧急联系机制，一旦出现意外立即处理
- **数据保护**：渗透测试过程中获取的敏感数据应立即报告并清理
- **恢复预案**：对测试可能造成的系统状态改变，预先制定恢复方案

#### 5.2.5 实地查验（Site Inspection）

**（1）方法概述**

实地查验是测评人员进入被测系统所在的物理环境，通过实地观察、测量、拍照等方式验证物理安全措施的落实情况。物理安全是网络安全的基础，实地查验是不可替代的测评方法。

**（2）查验项目详表**

| 查验大类 | 查验细项 | 查验方法和标准 | 常见问题 |
|----------|----------|---------------|----------|
| **物理位置选择** | 机房楼层位置 | 是否位于建筑的高层或地下室（防水患考量）、是否有窗户（防渗漏和防破坏） | 机房设在地下室（水患风险）或顶层（漏雨风险） |
| | 周边环境 | 是否临近公共区域、是否存在强电磁干扰源、是否靠近污染源 | 机房隔壁是公共卫生间（水管风险） |
| **物理访问控制** | 机房门禁 | 是否有机房门禁系统、门禁是否支持刷卡+密码/生物识别、是否有防尾随措施 | 门禁使用简单密码、门禁卡未绑定实名 |
| | 访客管理 | 是否有访客登记制度、访客是否全程陪同、是否有访客记录 | 访客可自由进入、无登记记录 |
| | 分区管理 | 机房是否按安全等级分区管理、不同区域是否有独立门禁 | 所有设备混放在一个房间 |
| **防盗窃防破坏** | 设备固定 | 核心设备是否固定在机柜中、机柜是否上锁 | 设备放置桌面未固定 |
| | 监控覆盖 | 视频监控是否覆盖机房出入口和主要通道、录像保存时间是否≥90天 | 监控有死角、录像仅保存30天 |
| | 报警系统 | 是否有入侵报警系统、报警是否接入24小时值守 | 报警无人响应 |
| **防雷击** | 防雷接地 | 是否有机房防雷接地、接地电阻是否达标（≤4Ω）、是否有定期检测记录 | 无防雷检测报告 |
| | 电源防雷 | 配电柜是否安装电源防雷器（SPD） | 配电柜无防雷器 |
| | 信号防雷 | 重要网络线路是否安装信号防雷器 | 室外网线直连无防雷 |
| **防火** | 消防系统 | 机房是否配备气体灭火系统（禁止水喷淋）、是否有烟感和温感探测器 | 机房用普通水喷淋系统 |
| | 防火材料 | 机房装修材料是否满足防火等级要求 | 使用易燃材料装修 |
| | 消防联动 | 消防系统是否与门禁联动（火灾时自动释放门禁） | 火灾时门禁锁死无法逃生 |
| | 消防检测 | 是否有消防设施定期检测记录 | 无年检记录 |
| **防水防潮** | 防水检测 | 机房内是否有水浸检测传感器、是否接入监控 | 无漏水检测 |
| | 空调排水 | 精密空调排水管是否畅通、是否有防漏水托盘 | 空调漏水无人知 |
| | 温湿度控制 | 是否有精密空调、温湿度是否在标准范围内（温度20-25°C，湿度40%-70%） | 使用普通舒适空调、温湿度超标 |
| **防静电** | 防静电地板 | 是否铺设防静电地板、接地是否有效 | 地板接地电阻超标 |
| | 防静电措施 | 是否有防静电手环、防静电工作台 | 操作人员无防静电措施 |
| **电力供应** | 双路供电 | 是否有两路市电输入、是否能自动切换 | 仅单路供电 |
| | UPS系统 | UPS容量是否满足需求、后备时间是否≥15分钟、是否有定期维护 | UPS电池老化、后备时间不足 |
| | 发电机 | 是否有备用发电机、燃料储备是否充足、是否定期测试 | 发电机长期未启动测试 |
| | 配电线路 | 电源线和通信线是否分开铺设、是否有防火线槽 | 强弱电线混在一起 |
| **电磁防护** | 电磁屏蔽 | 是否有机房电磁屏蔽措施（特别是涉密机房） | 无任何屏蔽措施 |
| | 线缆防护 | 关键线缆是否穿管保护、是否远离强电磁源 | 网线与电源线并行走线 |
| | 设备接地 | 所有设备是否可靠接地 | 部分设备无接地 |

**（3）实地查验操作要求**

- **准备充分**：提前准备查验清单、照相机（或手机）、测量工具（万用表、测温仪、测距仪等）
- **照片留证**：每个查验项至少拍摄1-2张照片作为证据，照片应包含拍摄时间和位置信息

- **安全第一**：进入机房应遵守机房管理规定，如穿戴防静电服、鞋套等
- **双人同行**：进入核心机房区域应有两人同行，一是安全需要，二是相互见证

### 5.5 问题确认

测评过程中发现的安全问题应与被测单位进行正式确认，这是确保测评结果客观公正的重要环节：

**问题确认流程**：

1. **问题梳理**：测评组对已发现的问题进行梳理和汇总，确保问题描述准确、依据充分
2. **问题沟通**：与被测单位召开问题确认沟通会，由测评组逐项说明发现的问题
3. **证据展示**：向被测单位展示相关的测评证据，包括截图、记录、照片等
4. **问题确认**：被测单位对每个问题进行逐一确认：
   - 认可的：签字确认
   - 有异议的：提出异议理由
5. **异议处理**：对存在异议的问题，进行补充测评和核实：
   - 误判的：纠正并说明原因

### 5.6 末次会议

现场测评工作全部结束后，应召开末次会议，标志着现场测评阶段的正式结束：

**会议议程**：

1. **工作总结**：测评组长总结现场测评工作的开展情况，包括完成了哪些测评项、投入了多少人天等
2. **问题通报**：向被测单位通报本次测评发现的主要问题概况（按严重程度和安全类别汇总），但不宣读具体细节（细节以问题确认单为准）
3. **后续安排**：说明后续报告编制的时间计划（通常需要10-20个工作日），以及被测单位需要配合的事项

7. **致谢**：感谢被测单位的配合和支持
8. **会议纪要**：形成末次会议纪要并由双方签字确认

## 六、报告编制活动

### 6.1 单项测评结果判定

#### 6.1.1 判定依据

单项测评结果判定是将实际测评结果与标准预期结果进行比对，判断该测评项的安全措施是否达到要求。判定的核心依据是 GB/T 28448-2019《信息安全技术 网络安全等级保护测评要求》中对各测评项的预期结果描述。

#### 6.1.2 判定结果分类

| 判定结果 | 含义 | 判定条件 | 对整体结论的影响 |
|----------|------|----------|-----------------|
| **符合** | 安全措施完全达到要求 | 实际结果与预期结果一致，能找到充分的证据证明安全措施有效 | 正面贡献 |
| **不符合** | 安全措施未达到要求 | 实际结果与预期结果存在明显差距，安全措施缺失或无效 | 负面影响，可能导致整体"不符合" |
| **部分符合** | 安全措施部分达到要求 | 实际结果与预期结果存在部分差距（如措施存在但不完善，或部分执行） | 轻微负面影响 |
| **不适用** | 测评项不适用于被测系统 | 系统中确实没有该测评项对应的应用场景（如无无线网络的系统） | 不影响整体结论，从基数中剔除 |

#### 6.1.3 判定原则

1. **证据充分原则**：每一个判定结论必须有充分的证据支撑。无证据支撑的判定结论无效。
2. **标准一致原则**：同一测评指标在不同测评对象上的判定标准应保持一致。不能对核心设备从严、一般设备从宽。
3. **独立判定原则**：每个测评指标独立进行判定，不应受其他指标判定结果的影响。
4. **双重确认原则**：对于"不符合"判定，尤其是高危问题的"不符合"，应有两人以上独立确认。
5. **基于事实原则**：判定应基于客观事实和证据，不应掺杂测评人员的主观推测。
6. **最小干预原则**：对边缘情况的判定，如果证据不充分，不应直接判定为"不符合"，而应进行补充测评。

#### 6.1.4 常见判定场景示例

| 场景描述 | 判定结果 | 理由 |
|----------|----------|------|
| 配置了防火墙访问控制策略，且策略严密度达到要求 | 符合 | 措施完备且有效 |
| 有防火墙但未配置任何访问控制策略 | 不符合 | 有设备但未使用 |
| 有防火墙但访问控制策略存在明显疏漏 | 部分符合 | 措施有但不够有效 |
| 系统部署在云平台，物理安全由云服务商负责 | 不适用 | 无相应的自管物理环境 |
| 配置了口令复杂度策略但未配置过期策略 | 部分符合 | 口令策略不完整 |
| 有安全审计日志但保存时间不足标准要求的6个月 | 部分符合 | 有审计但保存时间不足 |

### 6.2 单元测评结果判定

#### 6.2.1 单元划分

单元测评是将属于同一安全类或同一层面的测评结果进行汇总分析和判定。两种常见的单元划分方式：

**方式一：按安全类划分（推荐）**

| 单元编号 | 单元名称 | 包含内容 |
|----------|----------|----------|
| U01 | 安全物理环境 | 所有物理安全测评项 |
| U02 | 安全通信网络 | 所有通信网络安全测评项 |
| U03 | 安全区域边界 | 所有区域边界安全测评项 |
| U04 | 安全计算环境 | 所有计算环境安全测评项（主机+应用+数据） |
| U05 | 安全管理中心 | 所有安全管理中心测评项 |
| U06 | 安全管理制度 | 所有管理制度测评项 |
| U07 | 安全管理机构 | 所有管理机构测评项 |
| U08 | 安全管理人员 | 所有人员安全测评项 |
| U09 | 安全建设管理 | 所有建设管理测评项 |
| U10 | 安全运维管理 | 所有运维管理测评项 |

**方式二：按安全层面划分（用于整体分析）**

| 单元编号 | 单元名称 | 包含安全类 |
|----------|----------|-----------|
| 技术安全 | 技术层面 | U01-U05（物理+通信+边界+计算+管理中心） |
| 管理安全 | 管理层面 | U06-U10（制度+机构+人员+建设+运维） |

#### 6.2.2 单元判定方法

**（1）多对象汇总规则**

当同一测评指标应用于多个测评对象时，按照"短板原则"进行汇总：

- 所有测评对象均判定为"符合" → 该指标的单元汇总结果为"符合"
- 存在任一测评对象判定为"不符合" → 该指标的单元汇总结果为"不符合"
- 不存在"不符合"但存在"部分符合" → 该指标的单元汇总结果为"部分符合"
- 所有测评对象均为"不适用" → 该指标的单元汇总结果为"不适用"

**（2）单元符合性统计**

对每个单元进行定量统计分析：

```
单元符合率 = 符合项数 / （该单元总测评项数 - 不适用项数） × 100%
单元部分符合率 = 部分符合项数 / （该单元总测评项数 - 不适用项数） × 100%
单元不符合率 = 不符合项数 / （该单元总测评项数 - 不适用项数） × 100%
```

**（3）单元结果判定结论**

| 单元符合率 | 判定结论 |
|-----------|----------|
| ≥90% 且无高危不符合项 | 优 |
| 80%-89% | 良 |
| 70%-79% | 中 |
| 60%-69% | 较差 |
| <60% 或存在严重高危不符合项 | 差 |

### 6.3 整体测评结果判定

#### 6.3.1 整体测评的概念与方法

整体测评是在单元测评结果的基础上，进一步分析安全措施之间的关联性、互补性和整体效果，从系统全局角度综合评价安全保护能力。

**整体测评需要回答的核心问题**：

1. 各安全措施之间是否形成了协同的防护体系？
2. 单个单元的不足是否被其他单元的安全措施所弥补？
3. 系统的整体安全保护能力是否达到了对应等级的基本要求？
4. 存在的不符合项是否导致了不可接受的安全风险？

#### 6.3.2 安全措施间关系分析

| 关系类型 | 说明 | 分析方法 | 对判定的影响 |
|----------|------|----------|-------------|
| **互补关系** | 不同层面的安全措施相互补充，形成纵深防御 | 分析安全措施是否形成了多层防护体系 | 某一个层面的不足可能被其他层面弥补 |
| **关联关系** | 某项安全措施的正常运行依赖另一项措施 | 分析措施之间的依赖链路 | 基础措施不符合会导致依赖措施的实际效用降低 |
| **覆盖关系** | 某个通用安全措施覆盖了多项具体安全要求 | 检查统一安全平台的有效性 | 统一平台的有效部署可以同时满足多项要求 |
| **补偿关系** | 某项措施未达到标准要求，但其他措施提供了等效保护 | 评估补偿措施的等效性和有效性 | 补偿措施有效时可降低不符合项的影响 |

**互补关系分析示例**：

```
                         ┌─────────────────┐
                         │    应用安全      │ ← 应用层防护（WAF、代码安全）
                         └───────┬─────────┘
                                 │
                         ┌───────▼─────────┐
                         │    主机安全      │ ← 主机层防护（HIDS、基线加固）
                         └───────┬─────────┘
                                 │
              ┌──────────────────┼──────────────────┐
              │                  │                  │
    ┌─────────▼────────┐ ┌───────▼─────────┐ ┌─────▼───────────┐
    │   网络安全        │ │   数据安全       │ │   物理安全       │
    │（防火墙/IDS/IPS） │ │（加密/审计/DLP） │ │（门禁/监控/消防）│
    └──────────────────┘ └─────────────────┘ └─────────────────┘
```

分析要点：即使某一层存在部分不符合，只要其他层提供了充分的补偿保护，且整体保护能力达到预期，可不将该不符合列为"否决项"。

#### 6.3.3 整体判定原则与流程

整体测评结果判定应遵循以下流程：

```
单项测评结果 → 单元测评结果 → 安全措施关系分析 → 综合风险评价 → 整体结论
```

**综合风险评价要点**：

1. **不符合项风险评估**：对每个不符合项进行风险评估，分析其可能导致的最大安全影响
2. **组合风险分析**：分析多个不符合项组合可能产生的叠加效应
3. **补偿措施评估**：评估现有其他安全措施对不符合项的补偿效果
4. **残余风险判定**：综合评估系统的残余风险是否在可接受范围内

#### 6.3.4 整体判定结论

| 结论 | 条件描述 |
|------|----------|
| **基本符合** | 系统安全保护能力基本达到对应等级要求。各单元测评结果整体良好，存在少量不符合项但不影响系统的整体安全保护能力，且不符合项对应的安全风险在可接受范围内。通常情况下，综合符合率（排除不适用项后）应达到80%以上，且无未补偿的高危不符合项。 |
| **不符合** | 系统安全保护能力未达到对应等级要求。存在较多不符合项，或存在严重影响系统整体安全保护能力的高危不符合项且无有效补偿措施，系统的整体安全保护能力存在较大差距。 |

**判定为"不符合"的常见情况**：

- 存在多项高危不符合项且无法通过其他措施补偿
- 某个安全类的符合率极低（如<50%）且严重影响了整体防护
- 关键安全措施完全缺失（如无任何边界防护、无任何访问控制措施）
- 安全管理和技术措施严重脱节（有制度无执行）

### 6.4 测评报告编制

#### 6.4.1 报告编制要求

测评报告是测评工作的最终交付成果，也是被测单位了解自身安全状况和改进方向的最重要文件。报告编制应满足以下要求：

1. **格式规范性**：严格按照国家主管部门规定的等级保护测评报告模板进行编制。模板通常包括封面、摘要、被测系统描述、测评活动概述、测评结果、问题汇总、整改建议、测评结论等部分。
2. **内容完整性**：报告应包含所有要求的要素，不得遗漏。每个测评指标的测评结果都应在报告中体现。
3. **数据准确性**：报告中的数据、百分比、结论等必须基于实际测评证据，确保准确无误。任何数据的得出都应有据可查。
4. **结论客观性**：基于测评证据做出客观评价，不夸大、不缩小、不隐瞒。报告结论应经得起第三方复核。
5. **建议可行性**：提出的整改建议应具体、明确、具有可操作性，便于被测单位理解和实施。
6. **保密管理**：测评报告是包含系统安全漏洞和薄弱环节的敏感文件，应进行密级标识并妥善管理。

#### 6.4.2 报告结构详解

**封面**：报告编号、报告名称、被测单位名称、测评机构名称（加盖公章）、报告日期、密级标识

**版本控制页**：版本号、修订日期、修订人、修订内容说明

**第一章：报告摘要**
- 测评工作概述（测评时间、地点、人员、过程概要）
- 被测系统基本信息（名称、等级、用途）
- 测评范围和测评依据
- 主要测评发现（概述性的问题总结）
- 总体测评结论（基本符合/不符合）

**第二章：被测系统描述**
- 2.1 系统基本情况（名称、用途、服务范围、用户规模）
- 2.2 系统定级情况（安全保护等级、定级时间、备案情况）
- 2.3 网络拓扑和结构描述（附网络拓扑图）
- 2.4 主要设备/系统清单（按类别列表）
- 2.5 主要业务应用描述
- 2.6 安全措施概述（已部署的安全产品和管理措施）

**第三章：测评活动概述**
- 3.1 测评过程概要（四阶段简要说明）
- 3.2 测评对象选择（选择原则和最终确定的测评对象）
- 3.3 测评指标确定（指标的来源、总数、不适用项处理）
- 3.4 测评方法说明（使用了哪些方法、各种方法的适用范围）
- 3.5 测评工具清单（使用的工具及其用途）

**第四章：单项测评结果**（报告的核心章节之一）

- 4.2 安全通信网络测评结果
- 4.3 安全区域边界测评结果
- 4.4 安全计算环境测评结果
- 4.5 安全管理中心测评结果
- 4.6 安全管理制度测评结果
- 4.7 安全管理机构测评结果
- 4.8 安全管理人员测评结果
- 4.9 安全建设管理测评结果
- 4.10 安全运维管理测评结果

每个小节包含：测评结果汇总表（符合/不符合/部分符合/不适用统计）、不符合项详细描述

**第五章：单元测评结果**
- 5.1 各安全单元测评结果汇总
- 5.2 各单元符合率统计
- 5.3 单元测评分析

**第六章：整体测评分析**
- 6.1 安全措施间互补和关联关系分析
- 6.2 不符合项的综合风险分析
- 6.3 整体安全保护能力评价

**第七章：安全问题汇总**
- 7.1 问题清单（按安全类、严重程度排列）
- 7.2 问题统计（多维度统计图表）
- 7.3 重点问题风险分析

**第八章：整改建议**
- 8.1 整改总体建议
- 8.2 具体整改措施（按问题编号给出针对性建议）
- 8.3 整改优先级建议（高优/中优/低优，含建议完成时间）
- 8.4 整改验证方法建议

**第九章：测评结论**

**附件**
- 附件A：测评对象详细清单
- 附件B：测评指标-结果映射表
- 附件C：漏洞扫描详细结果

### 6.5 报告审核与交付

#### 6.5.1 多级审核流程

测评报告编制完成后应经过严格的多级审核，确保报告质量：

| 审核级别 | 审核角色 | 审核周期 | 审核重点 | 审核方式 |
|----------|----------|----------|----------|----------|
| 一级审核 | 测评组长 | 1-2天 | 技术内容正确性、数据准确性、判定合理性 | 详细审查 |
| 二级审核 | 项目负责人 | 1-2天 | 报告完整性、逻辑一致性、结论恰当性 | 全面审查 |
| 三级审核 | 技术负责人 | 1-2天 | 技术合规性、判定依据充分性、与其他标准的协调性 | 技术审查 |
| 四级审核 | 质量负责人 | 1天 | 格式规范性、过程合规性、文档完整性 | 形式审查 |
| 批准 | 机构负责人 | 1天 | 报告最终批准发布 | 签批 |

**审核不通过的处理**：任何一级审核不通过，报告退回修改。重大问题应召开技术讨论会确定修改方案。

#### 6.5.2 报告交付流程

- 报告经所有审核环节通过、机构负责人批准后，盖章装订
- 按照协议约定份数（通常3-5份）制作正式报告
- 报告交接应办理正式的《报告交付签收单》，双方签字确认
- 测评机构留存至少1份正式报告副本和全部过程文档
- 电子版报告（PDF格式）按要求交付或留存

## 七、测评过程管理

### 7.1 风险控制

#### 7.1.1 风险识别矩阵

测评过程中可能出现的风险需在项目启动时即进行识别和评估：

| 风险编号 | 风险类别 | 风险描述 | 发生概率 | 影响程度 | 风险等级 |
|----------|----------|----------|----------|----------|----------|
| R01 | 系统运行风险 | 漏洞扫描或渗透测试导致被测系统服务异常或中断 | 中 | 高 | 高 |
| R02 | 数据安全风险 | 测评过程中获取或接触到被测系统敏感数据 | 中 | 高 | 高 |
| R03 | 测评结果风险 | 由于抽样不充分或方法不当导致测评结论偏差 | 低 | 高 | 中 |
| R04 | 人员安全风险 | 测评人员在现场工作中的人身安全（如触电、跌落等） | 低 | 高 | 中 |
| R05 | 合规风险 | 测评活动超出授权范围或未遵守法律法规要求 | 低 | 高 | 中 |
| R06 | 沟通风险 | 与被测单位沟通不畅导致误解、冲突或工作延误 | 中 | 中 | 中 |
| R07 | 进度风险 | 配合不到位或突发情况导致测评进度严重滞后 | 中 | 中 | 中 |
| R08 | 设备风险 | 测评设备故障或丢失，影响测评工作 | 低 | 中 | 低 |
| R09 | 工具风险 | 测评工具版本过旧或功能失效，导致扫描结果不准确 | 低 | 中 | 低 |
| R10 | 人员变动风险 | 核心测评人员因故离开项目 | 低 | 高 | 中 |

针对上述风险，制定具体的控制措施：

| 风险编号 | 风险控制措施 | 责任人 | 实施时间 |
|----------|-------------|--------|----------|
| R01-系统运行 | ①制定应急预案；②在非业务高峰期进行有风险的操作；③在测试环境先验证后再用于生产；④限制并发扫描线程数；⑤实时监控系统运行状态 | 测评组长 | 贯穿测评全程 |
| R02-数据安全 | ①签署保密协议；②测评设备全盘加密；③测评数据脱敏后使用；④测评结束后彻底清理所有临时数据；⑤数据仅存储在受控设备上 | 项目负责人 | 贯穿测评全程 |

#### 7.1.3 应急预案

必须制定详细的应急预案，明确以下内容：

**紧急情况分类**：

| 紧急情况 | 判断标准 | 处理步骤 | 应急联系人 |
|----------|----------|----------|-----------|
| 被测系统服务中断 | 关键业务不可用 | ①立即停止当前测评操作；②通知被测单位运维人员；③协助排查问题原因；④配合恢复服务；⑤记录事件经过 | 被测单位运维负责人 |

| 人员安全事故 | 触电、受伤等 | ①立即采取急救措施；②拨打急救电话；③通知项目负责人和被测单位 | 项目负责人 |

**应急联系人清单**：应包含双方的项目负责人、技术负责人、紧急联系人的姓名、职务、电话、备用联系方式。

#### 7.2.2 文档管理细则

- **编号管理**：所有文档统一编号，格式为 [文档类型代码]-[日期]-[序号]

- **沟通记录**：正式沟通应形成记录，包括时间、地点、参与人员、讨论内容、结论和待办事项
- **信息对称**：确保双方对同一事项的理解一致，避免因信息不对称导致误解
- **及时反馈**：对被测单位提出的问题和关切及时响应和反馈
- **礼貌专业**：保持专业、礼貌的沟通态度，即使面对异议也应理性处理

## 八、不同保护等级的测评过程差异

### 8.1 各级测评过程特点对比

| 测评环节 | 第一级 | 第二级 | 第三级 | 第四级 |
|----------|--------|--------|--------|--------|
| 测评对象选择 | 抽选代表性对象 | 抽选为主，关键对象全查 | 关键对象全查，一般对象科学抽样 | 原则上全部检查 |
| 测评指标数量 | 约50项 | 约135项 | 约211项 | 约260项以上 |
| 访谈要求 | 与1-2名关键人员简单访谈 | 与主要岗位人员访谈 | 全面访谈，覆盖所有安全相关岗位 | 深度访谈，双人访谈，全面交叉验证 |
| 文档审查 | 审查核心制度文件 | 审查主要管理制度和记录 | 全面审查所有制度、规程和记录 | 全面深度审查，逐项验证执行效果 |
| 配置核查 | 关键设备手动检查 | 主要设备手动+工具辅助 | 全面配置核查，工具+人工结合 | 全覆盖+双人复核+渗透验证 |
| 工具检测 | 可选，基础漏洞扫描 | 基础漏洞扫描+端口扫描 | 全面漏洞扫描+Web扫描+渗透测试建议 | 全面扫描+渗透测试+代码审计+社会工程测试 |
| 实地查验 | 关键区域简单检查 | 主要区域检查 | 全面实地查验，全部拍照留证 | 全面查验+专业测量+第三方检测 |
| 测评周期 | 2-3个工作日 | 5-8个工作日 | 10-15个工作日 | 15-20个工作日以上 |
| 测评人员配置 | 2-3人 | 3-5人 | 4-7人 | 6-10人以上 |
| 报告审核 | 简化（2级） | 标准（3级） | 严格（4级） | 最严格（5级） |
| 测评频率 | 建议每两年至少一次 | 每两年至少一次 | 每年至少一次 | 每半年至少一次 |

### 8.2 各级判定标准的差异

| 判定维度 | 第一级 | 第二级 | 第三级 | 第四级 |
|----------|--------|--------|--------|--------|
| 符合性判定严格性 | 相对宽松 | 标准 | 严格 | 最严格 |
| "不符合"容忍度 | 容忍少量非关键不符合 | 容忍少量低风险不符合 | 关键项必须符合 | 基本不允许存在不符合项 |
| 整体结论标准 | 综合保护能力达标即可 | 大部分要求达标 | 关键要求必须达标 | 全方位要求达标 |
| 不适用项比例上限 | 无硬性限制 | ≤30% | ≤20% | ≤10% |

### 8.3 第三级测评典型场景

由于第三级是实践中最常见的测评等级，以下补充说明第三级测评的典型工作场景和特殊要求：

**典型场景1：中大型企业核心业务系统**

适用系统：ERP系统、OA系统、财务系统、CRM系统、HR系统等
测评重点：身份鉴别（特别是特权账户管理）、访问控制（特别是越权访问）、数据安全（特别是数据泄露防护）

**典型场景2：政府机关对外服务系统**

适用系统：政务服务平台、政府门户网站、电子政务系统
测评重点：安全区域边界防护（与互联网的隔离）、入侵防范（政府网站常被攻击）、安全审计（需要满足监管要求）

**典型场景3：关键信息基础设施中的系统**

适用系统：能源、交通、水利、金融等关键基础设施中的信息系统
测评重点：安全管理中心（集中管控）、安全运维管理（特别是应急响应和灾难恢复）、等保四级的部分增强要求也建议参考执行

### 9.3 2018版与2012版的主要变化

| 变化维度 | GB/T 28449-2012 | GB/T 28449-2018 | 影响分析 |
|----------|-----------------|-----------------|----------|
| 名称 | 信息安全技术 信息系统安全等级保护测评过程指南 | 信息安全技术 **网络安全**等级保护测评过程指南 | 与《网络安全法》保持一致，保护对象扩展为"网络" |
| 保护对象范围 | 仅信息系统 | 网络（含传统信息系统、云计算平台、移动互联系统、物联网、工业控制系统、大数据平台） | 测评范围大大扩展，需适配多种对象类型 |
| 测评方法 | 3种（访谈、检查、测试） | 5种（访谈、文档审查、配置核查、工具检测、实地查验） | 方法更细化，操作指引更清晰 |
| 扩展要求 | 无 | 增加云计算、移动互联、物联网、工控、大数据五类扩展安全要求 | 需针对不同对象类型选择适用的扩展要求 |
| 测评流程规范化 | 较简略 | 每个阶段都有明确的输入、输出、参与角色和评审要求 | 过程更规范，但工作量也相应增加 |
| 文档和记录要求 | 基本要求 | 全面强化，增加了大量过程文档和记录表格模板 | 文档管理工作量显著增加 |
| 质量管控 | 较少提及 | 引入多级审核机制和独立质量监督角色 | 提升了测评质量，但增加了管理层级 |
| 风险控制 | 简单提及 | 系统化的风险识别、评估和控制措施 | 更重视测评过程中的风险管控 |
| 沟通管理 | 未专门规定 | 建立了系统的沟通机制 | 有助于减少误解和冲突 |

## 十一、总结与展望

### 11.1 GB/T 28449-2018 核心价值

GB/T 28449-2018《信息安全技术 网络安全等级保护测评过程指南》是网络安全等级保护测评工作的"操作手册"，为测评机构和相关方提供了从测评准备到报告交付的完整过程指引。其核心价值体现在：

1. **标准化**：统一了全国等级保护测评工作的流程和方法，确保不同测评机构出具的测评结果具有可比性
2. **规范化**：通过明确的阶段划分、角色定义、文档要求，减少了测评工作中的随意性
3. **质量保障**：通过多级审核机制和过程文档管理，有效保障了测评工作的质量
4. **风险可控**：通过系统的风险识别和控制措施，降低了测评过程对被测系统的潜在影响
5. **可追溯性**：通过完整的证据链和过程记录，使得测评结果经得起检验

### 11.2 核心要点回顾

| 维度 | 核心要点 |
|------|----------|
| **测评流程** | 四阶段递进：准备→方案→实施→报告，每阶段有明确的输入、活动和输出 |
| **测评方法** | 五大方法并行：访谈、文档审查、配置核查、工具检测、实地查验，根据指标特点灵活选用 |
| **结果判定** | 三级判定递进：单项判定→单元判定→整体判定，从微观到宏观综合评价 |
| **过程管理** | 全过程管控：风险控制、文档管理、沟通管理贯穿始终 |
| **质量保证** | 多级审核：从自审到机构负责人批准，层层把关 |

### 11.3 实际工作建议

对于测评机构：

- 将本标准作为内部培训的核心教材，确保所有测评人员理解并遵循标准流程
- 建立基于本标准的质量管理体系文件，形成标准化的操作规程
- 定期对照标准进行内部审核，持续改进测评过程

对于被测单位：

- 了解本标准的流程要求，在测评前做好充分的配合准备
- 以本标准为参考，建立内部安全自查的工作流程
- 重视测评后的整改工作，将测评发现的问题作为安全建设的优先级依据

对于监管机构：

- 以本标准为参考，检查测评机构是否按规范流程开展工作
- 在监督检查中参考本标准的测评方法和判定逻辑
