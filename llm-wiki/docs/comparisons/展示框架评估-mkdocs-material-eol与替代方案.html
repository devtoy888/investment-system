二、Zensical（官方推荐）vs MkDocs Material

Zensical 是什么

  • 作者同源：Material for MkDocs 原作者团队，定位"下一代 SSG"。
  • 技术栈：Rust + Python，PyPI 包 zensical，支持 pip/uv/Docker；原生读取 mkdocs.yml，配置零改动。
  • 许可证：MIT（告别 sponsorware，改 Zensical Spark 商业支持）。
  • 阶段：Alpha，官方称"已兼容 Material，可开始构建"，目标先功能对等再超越。

相比 MkDocs Material 的优势

维度Zensical 优势核实状态
架构合并 MkDocs+Material 为单一栈，摆脱未维护依赖事实
构建速度差异引擎 ZRX，重复构建声称 4–5x 更快Alpha 未全发挥
搜索自研 Disco 引擎，排序/过滤/聚合更强设计成立
设计脱离 Material Design，更现代、易品牌化事实
扩展性规划 module/component system（Phase 2/3/4）路线图

⚠️ 对你的 Wiki 的致命缺口：wikilinks / obsidian-bridge

  • 你重度依赖 [[wikilinks]]（obsidian-bridge）+ Graphify 知识图谱。
  • Zensical 插件兼容清单（Tier1/Tier2）未包含 obsidian-bridge。
  • GitHub issue #174：wikilinks 兼容"已加入 backlog，调研用户需求"——尚未实现。
  • 含义：直接迁移会导致 [[wikilinks]] 失效、图谱链接断裂，需自研 module 或等官方支持。
结论：Zensical 是官方正统后继，长期最省心；但 Alpha + wikilinks 缺口使其不适合立即切换，建议作为 6–12 个月后的目标态跟踪。

三、其他优秀展示框架全量评估

1. Quartz（jackyzha0）最契合知识库

  • 定位：专为 Obsidian 风格 Markdown 的 SSG（Node.js/TS）。12.8k★，v5 活跃，MIT。
  • 原生支持：Wikilinks Backlinks Graph View Explorer、全文搜索、标签。
  • 迁移成本：你已是 Obsidian 兼容写法，概念几乎零摩擦，图谱开箱即得。
  • 风险：配置用 quartz.config.yaml，需重写 nav/主题；中文需适配；需 Node 构建环境。

2. VitePress（Vue 团队）

  • 18k★，活跃，MIT，被 52k+ 项目依赖。极快 HMR、美观、中文文档优秀。
  • 缺口：无 wikilinks/backlinks/graph 原生支持。适合"文档站"而非互联知识库。

3. Docusaurus（Meta）

  • React 驱动，~60k★，MIT，功能最全（版本化/i18n/博客/MDX）。
  • 缺口：无 wikilinks/backlinks/graph；React 生态偏重，与 Obsidian 写法差异大；过度工程。

4. Wiki.js（Requarks）

  • 基于 Git 的 Wiki 应用（Node.js），带 Web 后台、多用户、搜索。
  • 缺口：非 Obsidian 风格、无原生图谱；是"应用"非"静态生成器"，偏离 Agent 直写 Markdown 模式。

5. Astro Starlight

  • 定位：Astro 文档主题。现代、快、美观。
  • 优势：性能极佳、组件灵活、良好中文。
  • 缺口：无 wikilinks/backlinks/graph 原生支持；需自行搭互联层。
  • 适用：可作为"重定制文档站"底座，但需额外开发图谱。

6. Hugo（Go 语言 SSG）

  • 定位：Go 写的通用静态站点生成器，最成熟、最快之一。GitHub 89,007★（2026-07-17）、Apache 2.0、发布频繁、有 JetBrains/CloudCannon 公司赞助、活跃论坛。
  • 优势：工程稳健度高于 Quartz（公司赞助+大社区，非个人主导）；构建极快（数万页秒级）；内置 i18n，中文支持优秀；图片处理/JS 打包/Tailwind 管线完善。
  • 缺口：原生不支持 wikilinks/backlinks/graph——需第三方主题/插件（如 quartz-plus 提供 backlinks+local graph+CJK）。原生 Hugo 需接插件或把 wikilinks 转标准链接。
  • 对图谱管线：Graphify/ECharts 仍可用（框架无关）；wikilinks 需转标准链接或加插件，否则 build-graph.py 建边会断（同 Zensical 风险）。
  • 适用：若重视长期工程稳健性胜过"开箱互联"，Hugo + Obsidian 向主题是可选项；但对你"wikilinks+图谱零改动"目标，不如 Quartz 顺。

为什么不是直接上 Zensical

  • 你的核心价值是 [[wikilinks]] 互联 + 图谱，Zensical 当前不支持 wikilinks（#174 backlog），直接迁移破坏全站链接。
  • Zensical 处于 Alpha，生产站点不宜押注。
  • 强行迁移需自研 module 或批量转标准链接——成本 ≥ 切 Quartz，却得不到开箱图谱。

⚠️ Graphify 图谱管线的可移植性（关键补充）

初版评估把 Graphify 隐含归在 MkDocs Material 名下，这是误导。经核实，Graphify 是完全独立于展示层的管线。
  • Graphify 是什么：graphifyy（PyPI 0.9.18，2026-07-17 发版，活跃，MIT，支持 chinese 额外包，Graphify-Labs 出品）。tree-sitter 抽取 docs/**/*.md 的 nodes/edges → NetworkX + Leiden 社区检测 → 导出 graph.json / graph.svg / graph.html。
  • 与 MkDocs Material 解耦：渲染层是 ECharts（graph-viewer.v2.js 力导向图）+ 静态 SVG lightbox，框架无关。任何能托管静态资源 + 引 CDN 的框架都能继续用。
  • 结论 1：换 VitePress / Docusaurus / Astro / Wiki.js / Zensical 都不影响 Graphify 本身运行——只需托管 graphify-out/ 产物，图谱照常显示。"某框架无 graph"仅指无原生图谱，而非不能挂 Graphify。
  • 结论 2（分水岭在 wikilinks）：你的 build-graph.py 建边依赖 [[wikilinks]] 文本。
    • 迁 Quartz → wikilinks 原样保留 → Graphify 边完整、图谱零损失；且原生 graph view 可作补充。
    • 迁 Zensical（当前） → wikilinks 失效须转标准链接 → 若 Graphify 未同步改读标准链接，则图谱边会断，需改 build-graph.py（成本不低）。
    • 迁 其他无 wikilinks 框架 → 同理需改造抽取逻辑或保留兼容层。
  • 对方案影响：Graphify 独立性进一步夯实"中期选 Quartz"——唯一同时满足"展示层原生互联"和"图谱管线零改动"的框架。

Q2：Zensical 路线图与可介入迁移时机？

关键：Zensical 路线图刻意不给具体日期（官方 "Rather than promising concrete dates"）。信号明确：

  • 当前 Alpha，优先"头几个月消灭初始 bug + 奔向与 Material for MkDocs 功能对等"。
  • Phased transition：Phase 1 最大兼容 → Phase 2 module system → Phase 3 功能对等 → Phase 4 component system + CommonMark（Rust 解析器）。
  • wikilinks 缺口：issue #174 在 backlog 无排期；module 公开 API 先给 Zensical Spark（付费）用，稳定后才公开。
  • 你的 EOL 窗口：MkDocs Material EOL = 2026-11-05（约 3.5 个月）。

可介入时机：现在→EOL（2026-11）❌ 不宜迁（Alpha+wikilinks 未落地）；EOL 后 6–12 个月（约 2027 年中）⏳ 复评节点——看 Phase 2/3 进展与 #174 状态。
