3. Cloudflare D1 — 历史数据的基石

3.1 为什么需要D1

方案查询历史趋势随数据增长
❌ 纯JSONL全量读到内存越来越慢
❌ 纯R2 JSON每次下载整个文件流量浪费
❌ Python脚本每次触发服务器服务器有状态
✅ D1SQL索引查询毫秒级始终

D1 = Cloudflare的Serverless SQLite。免费5GB，当前系统1年约2MB，可用1000年。

3.2 D1 Schema（6张表）

-- 1. 持仓快照表（每日一条）
CREATE TABLE portfolio_snapshots (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  date TEXT NOT NULL,
  total_cost REAL, total_value REAL,
  total_profit REAL, profit_pct REAL,
  tech_ratio REAL, building_progress REAL,
  details TEXT  -- JSON: 各基金明细
);
CREATE INDEX idx_portfolio_date ON portfolio_snapshots(date);

-- 2. 基金净值历史表
CREATE TABLE fund_values (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  code TEXT NOT NULL, date TEXT NOT NULL,
  estimated_nav REAL, confirmed_nav REAL,
  change_pct REAL,
  UNIQUE(code, date)
);

-- 3. KOL信号表
CREATE TABLE kol_signals (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  source TEXT NOT NULL,
  weibo_id TEXT UNIQUE,
  created_at TEXT NOT NULL,
  content TEXT, direction TEXT,
  topics TEXT, parsed_fund_code TEXT
);

-- 4. 操作记录表
CREATE TABLE operations (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  date TEXT NOT NULL, fund_code TEXT NOT NULL,
  fund_name TEXT, action TEXT NOT NULL,
  amount REAL, reason TEXT,
  linked_signal_id INTEGER,
  result_pct REAL
);

3.3 同步策略：批量回写

步骤1：服务器cron采集 → 写入本地JSONL/CSV（现有流程不变） 步骤2：交易日16:20 → sync_to_d1.py 运行 步骤3：读取本地JSONL → 解析 → 批量INSERT到D1 步骤4：D1 API通过Worker暴露 → 前端查询

为什么不是实时写D1？

✅ 当前系统（JSONL/CSV）已经稳定运行

✅ 服务器不一定每次都能访问D1（网络波动）

✅ 本地文件是真相源，D1是查询副本

✅ 批量写入比逐条快100倍
